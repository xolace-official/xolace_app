# CEO Plan: Feedback Mechanism
Generated by /plan-ceo-review on 2026-05-13
Branch: feat(🎙️)feedback-mechanism | Mode: SELECTIVE EXPANSION
Repo: xolace-official/xolace_app

## Vision

### 10x Check
The ambitious version of this feature turns every user interaction into a labeled training signal. Not just "heavier was selected" but "heavier because the mirror said calm when I felt trapped." Not just "gave up" but "gave up because the AI kept reflecting sadness instead of the anger underneath it." Combined with the session content, this becomes a self-improving system: feedback → AI prompt refinement → better mirrors → fewer gave-ups → less heavier.

### Delight Opportunities
1. Feedback submission in settings shows a simple "Sent." confirmation — one word, no fanfare. Matches Xolace's tonal restraint.
2. Mirror miss field copy: "What was off?" (not "What did we get wrong?") — keeps it neutral, doesn't make users feel responsible for explaining AI failures.
3. Gave-up options framed with empathy: "The mirror kept missing" / "I found it myself" / "I just needed to express it" — none of these feel like critique of the user.
4. Heavier follow-up appears as a soft card, not a modal — never blocks forward navigation.
5. Throttle respects the user's emotional state: if they dismiss the heavier prompt without answering, the throttle does NOT trigger — we don't penalize silence.

## Accepted Scope (in this plan)

- **Convex `feedback` table** with types: `general`, `mood_heavier`, `mirror_miss`, `gave_up`
- **`convex/feedback.ts`**: `submit` mutation (requireAuth + validation), `canAskContextual` query (throttle check for mood_heavier), `canSubmitGeneral` query (rate limit: max 5 general submissions per 24h per profile — public app, needed at launch)
- **Settings screen**: "Send feedback" row → `FeedbackDialog` component (text area, max 1000 chars) → type: `general`
  - `FeedbackDialog` calls `canSubmitGeneral` query; when limit reached: submit disabled + copy "You've sent feedback today — come back tomorrow."
- **Session end (activity-variant)**: after "heavier" selection → `HeavierFeedbackPrompt` card → throttled 24h per profile → type: `mood_heavier`
  - 3 categorical options: "The mirror missed" / "Life is just heavy right now" / "Something else"
  - "Something else" → expands optional text field inline ("Anything you want us to know?")
  - `selectedOption` + optional `text` both stored in feedback table
  - All three options are tappable; prompt is skippable (no option required)
  - `sessionId` passed from `SessionEndScreen` as prop to `ActivityVariant`
- **Clarify state**: optional "What was off?" TextInput (max 100 chars) added after "Not quite" / "Say more" tap → submitted alongside clarification → type: `mirror_miss`
  - New props on `ClarifyState`: `sessionId: Id<"sessions"> | undefined`, `turnIndex: number`
  - Passed from `ReflectScreen` using `sessionId` and `turnsCount` from the reflection machine
  - `mirror_miss` is fire-and-forget — PostHog is NOT captured for this type (cannot await + fire-and-forget simultaneously; fire-and-forget wins)
- **Gave-up state**: `GaveUpFeedbackCard` (3 tappable options: mirror_kept_missing / figured_it_out / needed_to_vent) → type: `gave_up`
  - Fire-and-forget: card collapses immediately on tap regardless of mutation outcome; `console.error` on failure; no retry UX
- **Throttle rule**: Contextual prompts (mood_heavier) throttled to max 1 per 24h per profile. Throttle only triggers on answer, not on dismiss. Mirror_miss and gave_up are per-event (no time throttle — they're rare).
- **PostHog event** emitted for `general`, `mood_heavier`, and `gave_up` types: `feedback_submitted` with `type`, `has_text`, `has_option`. **`mirror_miss` skips PostHog** (fire-and-forget, cannot await mutation).
- **Data retention**: `convex/jobs/dataRetention.ts` and `convex/jobs/accountDeletion.ts` updated to delete all feedback records for a profile on data wipe or account deletion.

## Architecture

### New Convex Schema
```typescript
feedback: defineTable({
  emotionalProfileId: v.id("emotional_profiles"),
  type: v.union(
    v.literal("general"),
    v.literal("mood_heavier"),
    v.literal("mirror_miss"),
    v.literal("gave_up"),
  ),
  // Required for mirror_miss and gave_up (session is always active at those moments).
  // Optional for general (settings feedback has no session context).
  // Optional for mood_heavier (session end, session already completed).
  sessionId: v.optional(v.id("sessions")),
  // mirror_miss only: 0 = first clarify attempt, 1 = second (final).
  // Range: 0..MAX_TURNS-1 — if MAX_TURNS changes in reflection-reducer.ts, update this comment.
  turnIndex: v.optional(v.number()),
  text: v.optional(v.string()),            // general: max 1000 chars; mood_heavier: max 300 chars; mirror_miss: max 100 chars
  selectedOption: v.optional(v.string()),  // gave_up only: "mirror_kept_missing" | "figured_it_out" | "needed_to_vent"
  createdAt: v.number(),
})
.index("by_profile", ["emotionalProfileId"])
.index("by_profile_and_type_and_created", ["emotionalProfileId", "type", "createdAt"])
```

**Per-type required fields enforced in the `submit` mutation:**
- `general`: text required (non-empty after trim)
- `mood_heavier`: no required fields beyond type; text is optional; sessionId passed if available
- `mirror_miss`: sessionId required; turnIndex required (0 or 1 from reducer state); text required (non-empty after trim)
- `gave_up`: sessionId required; selectedOption required; text unused

### Data Flow
```
SETTINGS feedback (general):
  User taps "Send feedback" → FeedbackDialog opens
    → canSubmitGeneral query checked:
      → if limit reached: submit disabled + "You've sent feedback today — come back tomorrow."
      → if allowed: User types text (max 1000 chars)
    → Submit → feedback.submit({ type: "general", text })
    → requireAuth → validate length → canSubmitGeneral server guard → ctx.db.insert
    → posthog.capture("feedback_submitted", { type: "general" })
    → "Sent." confirmation → dialog closes

HEAVIER contextual (mood_heavier):
  User taps "heavier" in ActivityVariant
    → inline useQuery(api.feedback.canAskContextual)
      → checks last mood_heavier within 24h via index
    → if allowed: HeavierFeedbackPrompt card appears (below mood pills)
      → User answers OR dismisses
      → If answered: feedback.submit({ type: "mood_heavier", sessionId, text? })
        → throttle triggers (next ask blocked for 24h)
      → If dismissed: no record, throttle NOT triggered
      → Session end flow continues regardless

MIRROR MISS (mirror_miss):
  User taps "Not quite" / "Say more" on mirror state
    → clarify-state shows optional TextInput "What was off?"
    → User fills in text OR leaves empty
    → On "Let it out" (clarify submit):
      → if text non-empty: feedback.submit({ type: "mirror_miss", sessionId, turnIndex, text })
        (fire-and-forget — NOT awaited, PostHog NOT captured)
      → main clarification flow proceeds regardless

GAVE-UP (gave_up):
  Max clarification turns reached → gave-up state renders
    → GaveUpFeedbackCard appears (below empathetic message)
      → 3 options: mirror_kept_missing / figured_it_out / needed_to_vent
    → User taps option → feedback.submit({ type: "gave_up", sessionId, selectedOption })
      → card collapses immediately (submitted=true), regardless of mutation outcome
      → console.error on failure; no retry UX
    → Card disappears, navigation options remain available
```

### Throttle Architecture
Throttle is server-enforced in the mutation, not just the query. `canAskContextual` is the client-visible check (for UI show/hide), but `feedback.submit` also validates the throttle before inserting and silently no-ops if the throttle window hasn't expired. This prevents duplicates if two sessions end rapidly.

```
canAskContextual():  // hardcoded to mood_heavier; no type arg needed
  → requireAuth → get emotionalProfileId
  → query feedback table: .withIndex("by_profile_and_type_and_created", q =>
      q.eq("emotionalProfileId", profileId).eq("type", "mood_heavier"))
    .order("desc").first()
  → if no entry OR entry.createdAt < (Date.now() - 24 * 60 * 60 * 1000) → return true
  → else return false

canSubmitGeneral():  // hardcoded to general; no type arg needed
  → requireAuth → get emotionalProfileId
  → query feedback table: .withIndex("by_profile_and_type_and_created", q =>
      q.eq("emotionalProfileId", profileId).eq("type", "general")
       .gte("createdAt", Date.now() - 24 * 60 * 60 * 1000))
    .order("desc").take(6)                 // bounded: never reads more than 6 docs
  → if entries.length >= 5 → return false
  → else return true
  // NOTE: .collect().length is forbidden (convex_rules.md) — always use .take(N+1)

Client loading state: canAskContextual returns undefined while loading.
  → HeavierFeedbackPrompt renders null when result is undefined or false.
  → Never default-shows during load — avoids layout flash.
  → After a successful submit, the Convex query invalidates and returns false
    on next cycle — prompt disappears without an explicit dismiss call.

canSubmitGeneral loading state: when undefined, FeedbackDialog submit is enabled.
  → Server enforces rate limit regardless; client check is UX only.
  → Brief loading window is acceptable given the generous 5/day limit.

Throttle triggers on: successful mutation insert only.
Throttle does NOT trigger on: prompt dismissal, empty submit, mutation error.

Night mode: ActivityVariant has an existing `!isNight` guard around the entire
  mood selection block. HeavierFeedbackPrompt is nested inside that block —
  it never appears during night sessions. No additional guard needed.

GaveUpFeedbackCard: unmounts naturally when user navigates from gave-up state.
  No persistence needed. Tapping an option collapses the card immediately (set
  submitted=true local state, render null), regardless of mutation outcome.
  User can proceed without tapping. console.error on mutation failure; no retry.

When mutation server-throttle fires (race condition):
  → mutation returns null (no-op) rather than throwing.
  → client treats null as success — no error state shown to user.
```

### turnIndex Values
`turnIndex` for `mirror_miss` is 0-indexed and maps to `turnsCount` from the reflection machine at the moment the user taps clarify:
- Turn 0: first "Not quite" / "Say more" tap (`turnsCount == 0`)
- Turn 1: second (max) clarification attempt (`turnsCount == 1`)
Never exceeds 1 given MAX_TURNS = 2.

### Data Retention
The `feedback` table uses `emotionalProfileId` as its key — the same ID cleaned up by account deletion and data wipe. Both jobs must be updated:

- `convex/jobs/dataRetention.ts`: add feedback record deletion when profiles are wiped
- `convex/jobs/accountDeletion.ts`: add feedback record deletion alongside other profile data

Pattern (same as other tables with `by_profile` index):
```typescript
const feedbackRecords = await ctx.db
  .query("feedback")
  .withIndex("by_profile", q => q.eq("emotionalProfileId", profileId))
  .take(100);
for (const record of feedbackRecords) {
  await ctx.db.delete(record._id);
}
// Schedule continuation if more records exist
```

### mirror_miss Error Handling
`mirror_miss` submission is fire-and-forget. On Convex error:
- Client: `console.error("mirror_miss feedback failed", err)` — visible in dev builds
- Convex: standard server-side error logging
- No retry. No user-facing error state. Clarification flow is never blocked.
- PostHog: NOT captured for mirror_miss (fire-and-forget cannot await mutation result)

### PostHog Event Spec
PostHog capture is called client-side after `await mutation(...)` succeeds. If the mutation throws, PostHog is not called. If the mutation returns null (no-op throttle), PostHog is also not called — no false positive "feedback_submitted" events.

**`mirror_miss` is excluded from PostHog** — fire-and-forget contract requires not awaiting the mutation. Awaiting would block the clarification flow.

```
feedback_submitted:
  type: "general" | "mood_heavier" | "gave_up"    // mirror_miss excluded (fire-and-forget)
  has_text: boolean
  has_option: boolean  // for gave_up and mood_heavier "Something else"
  // No session_id or user ID sent to PostHog — external analytics system,
  // privacy model requires no linkable identifiers.
```

### mirror_miss Concurrent Mutation Lifecycle
The text value for `mirror_miss` is captured into a local variable before the clarification mutation fires. Even if the component unmounts during the `processing` state transition, the captured string is already in closure scope and the fire-and-forget `feedback.submit` call completes independently. The feedback mutation is dispatched synchronously on button press, not awaited.

```
// In clarify-state.tsx:
const handleSubmit = () => {
  const capturedFeedbackText = feedbackText.trim(); // capture before dispatch
  dispatch({ type: 'SUBMIT_CLARIFY', text: clarifyText });
  if (capturedFeedbackText) {
    submitFeedback({ type: 'mirror_miss', sessionId, turnIndex, text: capturedFeedbackText });
    // submitFeedback is the Convex mutation — not awaited, PostHog not captured
  }
};
```

### `useHeavierFeedback` — Not a Hook
The implementation is inline in `activity-variant.tsx`: a `useQuery(api.feedback.canAskContextual)` call at the component level, and conditional rendering of `HeavierFeedbackPrompt` based on `selectedMood === 'heavier' && canAsk === true`. No separate hook file needed.

## Key Implementation Decisions

1. **No feedback table for anonymous users** — requireAuth in all mutations, emotional_profile is the key
2. **Throttle on answer, not dismiss** — critical for user trust; silence is not feedback
3. **Mirror miss is fire-and-forget** — never blocks the clarification flow, error is silent (logged); PostHog skipped
4. **Gave-up feedback is optional and fire-and-forget** — card collapses on tap regardless, no retry UX
5. **Settings feedback has a 1000-char limit** — enforced both client (disabled submit) and server (validator)
6. **Feedback records deleted on account deletion** — privacy-first: no orphaned user-linked records post-wipe
7. **canSubmitGeneral uses bounded count** — `.take(6)` never reads more than 6 docs; `.collect().length` is forbidden
8. **sessionId flows via props** — SessionEndScreen → ActivityVariant; ReflectScreen → ClarifyState (from reflection machine)

## Critical Test Invariants

The following invariants are product decisions that MUST be tested — they won't be obvious from implementation alone.

```
THROTTLE INVARIANTS (feedback.canAskContextual + mood_heavier type):
  T1: First-time heavier selection → prompt shows (no prior entries)
  T2: Heavier → answered → same session or next session within 24h → prompt hidden
  T3: Heavier → DISMISSED (no answer) → another heavier within 24h → prompt SHOWS again
        (dismiss does NOT trigger throttle — this is the critical invariant)
  T4: Heavier → answered → 25h later → prompt shows again

SUBMISSION INVARIANTS:
  T5:  mirror_miss with empty text → no feedback record created
  T6:  mirror_miss with text at turn 0 → record with correct sessionId + turnIndex=0
  T6b: mirror_miss with text at turn 1 → record with correct sessionId + turnIndex=1
  T7:  gave_up option tap → card collapses (submitted=true) → record created
  T8:  gave_up no tap → navigate away → no record created
  T9:  general feedback with empty text → server mutation never called
  T10: canAskContextual returns undefined (loading) → prompt hidden (not shown)

RATE LIMIT INVARIANTS (feedback.canSubmitGeneral + general type):
  T11: 4 general submissions in 24h → canSubmitGeneral returns true (submit allowed)
  T12: 5 general submissions in 24h → canSubmitGeneral returns false (limit reached)

SERVER VALIDATION INVARIANTS:
  T13: mirror_miss mutation called without sessionId → mutation throws (per-type validation)
  T14: gave_up mutation called without selectedOption → mutation throws (per-type validation)

PROGRESSIVE DISCLOSURE INVARIANTS:
  T15: HeavierFeedbackPrompt: tap "Something else" → optional text field appears
  T16: HeavierFeedbackPrompt: "Something else" + text → mutation called with
         selectedOption="something_else" AND text included in payload
```

## Reviewer Concerns (unresolved after 3 spec-review iterations)

1. **Index field order** — Verify `["emotionalProfileId", "type", "createdAt"]` is queried in exactly this order in `canAskContextual` and `canSubmitGeneral` (two `.eq()` calls then `.order("desc")`). Convex requires equality fields before range/order fields. This pattern is correct as designed; confirm during implementation.
2. **`canAskContextual` must declare `args: {}`** — All Convex query functions require an explicit args validator, even if empty. Omitting it causes a runtime error.
3. **`sessionId` availability in clarify state** — Clarify state is only reachable after session initiation + mirror confirmation attempt; `sessionId` from `useSession()` is guaranteed to be non-null by the time `mirror_miss` fires. Document this invariant as a comment in clarify-state.tsx.
4. **mirror_miss concurrent mutation ordering** — `sessionId` is captured in closure scope before `dispatch()` is called, so even if state transitions between the two calls, the captured ID is stable. Confirm this in implementation.

## Design Specifications

Added by `/plan-design-review` on 2026-05-14.

### FeedbackDialog (settings)

**Placement:** New `SettingsSection` titled **"Support"**, positioned between "Your Data" and the "Log out" row.

**State machine:**
- `isSaving=false, sent=false` (default): Send button enabled when `text.trim().length > 0`, disabled otherwise (`isDisabled`)
- `isSaving=true`: Send button shows `isLoading={true}`; textarea non-editable; Cancel disabled
- `sent=true`: Button label shows "Sent." (or `<Button isLoading={false}>Sent.</Button>`); `setTimeout(1500, () => onOpenChange(false))` closes dialog
- `error` (string | null): Inline error below textarea (`text-xs text-danger/80 mt-2`), using `extractErrorMessage` pattern from `SpaceNameDialog`
- Rate-limit (`canSubmitGeneral === false`): Textarea remains editable; Submit `isDisabled={true}`; copy below textarea: `text-xs text-foreground/50 mt-2` — "You've sent feedback today — come back tomorrow."; Cancel still works

**Character counter:** `text-xs text-foreground/30 text-right mt-1` — displays "N / 1000" below the TextArea. Updates as user types.

**Submit button:** Label "Send" (not "Save"). `variant="primary" size="sm"`.

### ActivityVariant

**ScrollView wrapper:** Wrap the entire `ActivityVariant` return in a `ScrollView` with `showsVerticalScrollIndicator={false}`. Remove `justify-center` from the inner `View`. Use `contentContainerStyle={{ flexGrow: 1, paddingVertical: 24 }}` to preserve breathing room at top/bottom on short content.

**HeavierFeedbackPrompt position:** Render immediately after the mood pill row (inside the `!isNight` block), before the contribution section. Conditional on `selectedMood === 'heavier' && canAsk === true`.

### HeavierFeedbackPrompt

**Card structure:** Mirrors `ReachFeedbackCard` card container:
```tsx
<Animated.View
  entering={FadeInDown.delay(400).duration(400)}
  exiting={FadeOut.duration(200)}
>
  <View className="mx-5 mt-4 p-4 rounded-2xl bg-surface border border-overlay/20">
    {/* prompt label */}
    {/* vertical option stack */}
    {/* "Something else" expanded field (conditional) */}
  </View>
</Animated.View>
```

**Option layout:** Vertical stack (`gap-2`), full-width buttons — NOT a `flex-row`. Each option:
```tsx
<Pressable
  onPress={() => handleOption(key)}
  accessibilityLabel={label}
  className="w-full py-3 rounded-xl bg-accent/10 items-center active:opacity-70"
>
  <AppText className="text-sm font-medium text-accent">{label}</AppText>
</Pressable>
```

**"Something else" expansion:**
- On tap: `setShowTextField(true)` → `Animated.View entering={FadeIn.duration(200)}` renders a `TextInput` inline below the option row
- Placeholder: `"Anything you want us to know?"`
- Max chars: 300 (matches plan schema)
