## Vision

### 10x Check
The Bridge becomes emotional communication infrastructure. Not just "compose a text" —
"help two people have a conversation they've been avoiding." The Bridge draft includes
suggested follow-up questions based on the emotional register. The recipient gets a deep-link
landing page with context: "Someone who cares about you wrote this after a reflection." The
Bridge stops being a share feature and becomes a relationship repair surface. A session in
Xolace initiates a two-person emotional conversation rather than a one-person relief valve.

## Implementation Clarifications (resolved pre-build)

### mirrorText null path
Bridge card must NOT render when `mirrorText` is null. Guard condition:
`mirrorText != null` evaluated before rendering `<BridgeCard />` in both ExitVariant and ActivityVariant.
Rationale: exit-variant sessions are exactly the sessions most likely to have no confirmed Mirror.
An empty pre-fill TextArea would be confusing; a crash is unacceptable. The Bridge is only
meaningful when there is a Mirror to translate.

### gestureEnabled scope
`gestureEnabled: false` applies to **Screen 2 only** (draft review), NOT Screen 1 (recipient context).
Screen 1 has no typed content; back-swipe to session-end is safe and expected.
Screen 2 has a typed draft the user may have edited; accidental back-swipe would destroy that work.
Pattern matches `sit-with-this` and `peer-reflections` which disable gestures to protect active state.

### useSessionEnd hook pre-step
`useSessionEnd` currently exposes `distilledText` but NOT `mirrorText`. Before the Bridge card
can pass `mirrorText` as a route param, the hook must be extended:
- In `src/features/session-end/hooks/use-session-end.ts`: add `mirrorText: session?.mirrorText ?? null` to the return object.
- This is a required pre-step before building the Bridge card. Do it in its own commit.

### Route structure
Single route file `src/app/(protected)/trusted-bridge.tsx` with local `step` state:
`type BridgeStep = "recipient" | "draft"`. Both screens render within the same route — no nested routes.
`mirrorText` is passed via typed route params from session-end. Expo Router typed routes
(`href: { pathname: "/(protected)/trusted-bridge", params: { sessionId, mirrorText } }`)
handle this cleanly. Note: `mirrorText` can be up to ~500 chars; Expo Router URL params
are safe up to OS URL limits (~2000 chars). If mirrorText exceeds safe length, pass only
`sessionId` and query mirrorText on Screen 1 via `useQuery(api.sessions.getMirrorText, { sessionId })`.

### Ownership validation (Stage 2 — correcting design doc pseudocode)
The design doc's `validateSessionOwnership` internalQuery checking `session.tokenIdentifier` is wrong —
that field does not exist on session documents. The correct pattern:
1. In `requestBridgeDraft` action: call `ctx.auth.getUserIdentity()` directly (actions can't use requireAuth)
2. Wrap `requireSessionOwnership()` from `convex/lib/auth.ts` in a thin `internalQuery` with NO `tokenIdentifier`
   parameter — `requireSessionOwnership` already derives auth identity from `ctx.auth` internally:
   ```typescript
   export const verifyOwnership = internalQuery({
     args: { sessionId: v.id("sessions") },
     handler: async (ctx, args) => {
       await requireSessionOwnership(ctx, args.sessionId);
     },
   });
   ```
   Call via `ctx.runQuery(internal.ai.bridge.verifyOwnership, { sessionId: args.sessionId })`.
   Note: the action also calls `ctx.auth.getUserIdentity()` directly for the null-check before calling
   `verifyOwnership`. This results in two `getUserIdentity()` calls per request — intentional and safe.
   Convex caches the identity within a request. The null-check in the action is belt-and-suspenders;
   `requireSessionOwnership` inside `verifyOwnership` will also throw "Not authenticated" if identity
   is null. Both guards serve different audiences (fast-fail in action vs. ownership enforcement in query).
3. `requireSessionOwnership` validates via `session.emotionalProfileId !== profile._id` — the correct check.
4. For the Anthropic call in `generateBridgeDraft`: extract the call into a shared async helper function
   (e.g., `_generateDraft(mirrorText, recipientName, relationship)`) and call it directly from
   `requestBridgeDraft` — do NOT use `ctx.runAction(internal.ai.bridge.generateBridgeDraft, args)`.
   Action-from-action is only justified when crossing runtimes (V8 → Node). The Anthropic HTTP call
   runs fine in the default Convex runtime; no Node built-ins required. Nested action call adds scheduling
   overhead with no benefit.
The design doc pseudocode should be discarded; this is the authoritative pattern.

### gestureEnabled implementation pattern
`trusted-bridge` is a single-route file with local `step` state — there is no separate "Screen 2 route"
to register with `gestureEnabled: false` in `_layout.tsx`. Do NOT set gestureEnabled: false at layout
registration time (that would disable it for Screen 1 as well). Instead, toggle it dynamically inside
the component:
```typescript
const navigation = useNavigation();
useEffect(() => {
  navigation.setOptions({ gestureEnabled: step !== "draft" });
}, [step]);
```
This enables gestures on Screen 1 (step === "recipient") and disables them on Screen 2 (step === "draft").
The `_layout.tsx` still needs to register the route, but with default options only.

### activity-variant.tsx extraction
`activity-variant.tsx` is 255 lines. Before adding BridgeCard, extract either `ReachFeedbackCard`
or `HeavierFeedbackPrompt` into their own import-only component files to bring the main
file under 200 lines. Budget ~1 hour. Do this extraction as a separate commit before the
Bridge card addition.

### Relationship selector in Stage 1
The relationship selector (parent/partner/friend/sibling) has no backend consumer in Stage 1
(no AI prompt call). Collect the selection client-side for analytics context only — include
`recipient_relationship: string | null` on the `bridge_opened` or `bridge_shared` PostHog event.
Do not pass it to any backend call in Stage 1.

## Accepted Scope from 11-Section Review (D6–D10)

### D6 — Error paths in Stage 1
Both error paths fixed in Stage 1:
- `Share.open()` must be wrapped in try/catch. User-cancelled share (user presses cancel on share sheet)
  throws in some platforms; uncaught throws crash Screen 2. Catch silently — cancelled send is valid.
- URL param length guard: before navigating to `/(protected)/trusted-bridge`, check
  `mirrorText.length > 500`. If so, pass only `{ sessionId }` as route param and query mirrorText
  on Screen 1 via `useQuery(api.sessions.getMirrorText, { sessionId })`. This prevents OS URL
  truncation on very long Mirror outputs.

### D7 — CTA disabled when recipient name empty
Primary CTA "Write this together" on Screen 1 is DISABLED when `recipientName.trim() === ""`.
Rationale: the draft (Stage 1 Mirror text, Stage 2 AI) is not personalized without a name. An
empty-name share is worse than no share — the Mirror text sent as-is reads as a therapist note.
Visual state: button in muted/disabled style, not hidden. User sees the field needs filling.

### D8 — bridge_dismissed analytics event
Add `bridge_dismissed` PostHog event for funnel completeness. Fires on back navigation from
either screen. Payload: `{ step: "recipient" | "draft" }`.
This makes the funnel analyzable: tapped → opened recipient (bridge_opened) → dismissed at recipient vs
dismissed at draft vs completed share (bridge_shared). Without bridge_dismissed, drop-off is invisible.

### D10 — Suppress SessionEndNotifNudge when Bridge eligible
`SessionEndNotifNudge` uses `className="absolute inset-0 bg-background"` — full-screen overlay
with 900ms delay. It will cover a rendered BridgeCard.
Fix: pass `hideBridgeOverlay={mirrorText != null}` prop to `SessionEndNotifNudge`, or use a
`suppressWhen` context value. The simplest implementation: SessionEndScreen already knows
`mirrorText` (after `useSessionEnd` pre-step). Pass a boolean prop:
```typescript
<SessionEndNotifNudge suppressWhen={mirrorText != null} />
```
Inside `SessionEndNotifNudge`: if `suppressWhen` is true, render null (skip the overlay entirely).
This does not affect the notif nudge's normal behavior for sessions without a Mirror.

## Outside Voice Integration (cross-model audit — Claude subagent)

8 issues surfaced. All resolved. Key ones:

**CRITICAL (caught by outside voice, missed by primary review):**
- `SessionEndNotifNudge` absolute overlay buries BridgeCard — resolved via D10 above.
- mirrorText evaporation concern: outside voice flagged risk of mirrorText becoming null between
  card render and tap. Verified non-issue: `completePath` fires only on dismiss/haveMore tap,
  NOT at Bridge card tap. mirrorText is stable in the route param after navigation.

**Minor items actioned without AskUserQuestion:**
- `useNavigation` import: use `import { useNavigation } from "@react-navigation/native"` — NOT from
  `expo-router`. `useRouter` does not expose `setOptions`. This is the correct Expo Router pattern.
- `first_sentence_changed` capture timing: compute the baseline first sentence at Screen 2 mount
  (store in a `useRef`), NOT at Share tap time. If user edits the first sentence then taps Share,
  capture-at-share-time would see the edited version and always return `false`. Capture-at-mount
  preserves the original for comparison.
- Back navigation affordance: Screen 2 has gestureEnabled: false. User needs an explicit back button.
  Add a header back button ("← Back") or inline "< Edit recipient" link above the TextArea. Without
  it, Screen 2 is a trap on devices without physical back buttons (iOS).
- Session eligibility context: exit-variant sessions are included in Stage 1, but many will have
  no mirrorText (they exited before Mirror confirmation). The 8% tap-rate success criterion is
  measured against eligible sessions (mirrorText != null), not all sessions. Analytics query must
  filter accordingly when evaluating Stage 1.
