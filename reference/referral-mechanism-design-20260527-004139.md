## Problem Statement

Xolace needs a word-of-mouth growth mechanism. Users who have processed something meaningful at the campfire want to share it with friends who might also need it — but the product has no way to track or reward that behavior. A referral code system creates an explicit, trackable word-of-mouth loop while preserving Xolace's zero-PII, zero-social-graph privacy architecture.

## Demand Evidence

Users are already sharing Xolace organically (WhatsApp sharing of quotes confirms ambient demand). The referral system converts that untracked behavior into a structured, measurable growth loop. No direct user request for referral codes — this is a proactive growth feature. The decision to build it before mirror sharing is based on build speed (referral codes are a fraction of the scope), not relative demand signal.

## Status Quo

Currently: zero acquisition infrastructure. Users who want to share Xolace send the App Store link manually. No tracking, no reward, no contextual entry point for the referee. The person who receives the link has no frame for why someone is sending them to a mental wellness app.

## Target User & Narrowest Wedge

**Referrer:** active Xolace users who have processed something real and want to share. The narrowest wedge: a referral code in settings + a "know someone who might need this?" callout at session end. Zero friction — just a copy/share button.

**Referee:** someone the referrer cares about who might be struggling. Entry: skippable code entry screen post-auth. In-app framing never reveals the referrer's identity. The app says: "Someone who cares about you shared this with you."

## Constraints

- Zero-PII architecture: users table has no name, no email. Referral codes work without usernames.
- No visible social graph within the app: referee never sees referrer's name or identity.
- MVP is app-only rewards (no physical merch). Merch deferred to v2; when shipped, shipping addresses collected via external redemption link — never stored in Xolace backend.
- Referral linkage tracked at emotional_profile ID level only (no auth identity exposed in referrals table).
- Code format: 6-character uppercase alphanumeric (e.g., XO7K2M). System-generated at account creation.

## Premises

1. Referral codes ship before mirror sharing — faster build, starts word-of-mouth immediately.
2. System-level identity linkage (referrer profile → referee profile), zero visible identity within the app.
3. Explicit referral point counter visible to user + milestone unlocks (themes, voice vent time, mirror tones).
4. Physical merch deferred to v2.
5. Every user gets a unique referral code at account creation.
6. Entry points: settings screen, session end screen.
7. Code entry: post-auth, before first session, skippable.

## Approaches Considered

### Approach A: Minimal (code + count on users, no junction table)
Add `referralCode`, `referralCount`, and `referredBy` to the users table. One mutation. No new tables. Fastest to ship but no audit trail, no fraud prevention, and will require schema migration when merch v2 lands.

### Approach B: Full junction table (SELECTED)
`referralCode` on users table. New `referrals` junction table tracking each (referrerProfileId, refereeProfileId, status, createdAt). Preferences gets `referral.unlockedTiers`. Count denormalized on `emotional_profiles` (same field as `sessionCount`). Same pattern as `reflection_resonances` — identical junction table structure. Full analytics + fraud prevention + zero schema migration for merch v2.

### Approach C: Deferred entry UX (architecture same as B)
Code entry prompt appears after first completed session instead of post-auth. More brand-coherent emotionally, but risks users not having the code handy by then.

## Recommended Approach

**Approach B.** Junction table is the complete design. Marginal cost over A is minimal (CC compresses this to ~15 extra minutes). Benefits: fraud prevention via unique (referrer, referee) pair, analytics, zero migration when physical merch ships.

### Schema changes

**`users` table additions:**
```
referralCode: v.optional(v.string())  // 6-char uppercase alphanumeric, generated at account creation
```
Must be `v.optional` — existing user documents do not have this field; adding a required `v.string()` will break Convex schema validation against all existing rows. New users created after this deploy will always have the field set (generated in `getOrCreate`). Existing users will have `undefined` — `getMyReferralInfo` must handle this: return `{ code: profile.referralCode ?? null, ... }` and the Settings UI should show a loading/generating state if code is null (or trigger a one-time backfill mutation on first load — see Dependencies).

Note: index required for code lookup — add `.index("by_referral_code", ["referralCode"])` to the users table definition.

**`emotional_profiles` table additions:**
```
referralCount: v.optional(v.number()),  // denormalized count of completed referrals (starts 0)
referredByProfileId: v.optional(v.id("emotional_profiles")),  // profile that referred this user
```
Storing `referredByProfileId` on `emotional_profiles` (not `users`) is consistent with the constraint that referral linkage is tracked at the emotional-profile level, not the auth-identity level. It also matches the existing pattern where `sessionCount`, `currentStreak`, and `dominantEmotionTags` all live on `emotional_profiles`.

**New `referrals` table:**
```
referrals: defineTable({
  referrerProfileId: v.id("emotional_profiles"),  // who shared
  refereeProfileId: v.id("emotional_profiles"),    // who joined using the code
  referralCode: v.string(),                        // the code used (for audit)
  status: v.union(v.literal("pending"), v.literal("completed")),
  // "pending" = code entered, "completed" = referee completed first session
  createdAt: v.number(),
})
  .index("by_referrer", ["referrerProfileId"])
  .index("by_referee", ["refereeProfileId"])
  .index("by_code", ["referralCode"])
```

**`preferences` table addition:**
```
referral: v.optional(v.object({
  unlockedTiers: v.array(v.string()),
  // e.g. ["first_referral", "three_referrals", "five_referrals"]
}))
```
Note: `referralCodePromptDismissed` is a Zustand local-store flag only — do NOT add it to this schema.
Note: `referralCount` on `emotional_profiles` uses `v.optional(v.number())`. Existing rows will have `undefined` — always read as `profile.referralCount ?? 0`. No backfill migration needed.

**Ember theme (existing stub — unlock, don't create from scratch):**
`src/lib/themes.ts` already defines Ember as `{ id: 'ember', tier: 'premium', available: false }`. The existing architecture has `ThemeTier = 'free' | 'premium'` and `FREE_THEMES` filters by `tier === 'free'`. The `AppearanceScreen` renders only `FREE_THEMES`.

Changes required:
1. Add `'earned'` to `ThemeTier` union in `themes.ts`: `type ThemeTier = 'free' | 'premium' | 'earned'`
2. Change Ember's entry: `tier: 'earned'`, `available: true`
3. Create `src/themes/ember.css` — warm amber/orange, both `@variant ember-light` and `@variant ember-dark` blocks, all required CSS variables. Register in `metro.config.js` `extraThemes` and `src/context/app-theme-context.tsx` `ThemeName` union.
4. Update `AppearanceScreen` to render `EARNED_THEMES` (add `export const EARNED_THEMES = THEMES.filter(t => t.tier === 'earned')`) below `FREE_THEMES`. Each earned theme card checks `unlockedTiers.includes("first_referral")` — if not earned, render locked state with "Earn with 1 referral" label; if earned, render normally. Note: the premium themes section in `AppearanceScreen` is currently entirely commented out — leave it commented. Add the earned section as new JSX below the free section, do not uncomment the premium block.

**Ember color palette** (derive from existing stub in `themes.ts`): dark variant uses `bg: '#100804'`, `accent: '#e8834a'` as anchors. Map these to CSS variables following the same contrast-ratio pattern as the `quiet` theme: `--background` → bg, `--accent` → accent, `--foreground` → readable contrast against bg, `--surface` → slight lift from bg, `--overlay` → semi-transparent bg, `--border` → muted accent. ### Convex functions

- `generateReferralCode()` (internal utility, `convex/users.ts`) — called from `users.getOrCreate` **only in the new-user branch** (after `ctx.db.insert`, not in the early-return branch for existing users). Generates a unique 6-char uppercase alphanumeric code. Queries `users.by_referral_code` to check for collision before returning; retries up to 5 times. After 5 failed retries, throws a Convex error — `getOrCreate` propagates this as a mutation failure visible to the client as a failed auth. At current user volumes (36^6 ≈ 2.2B possible codes) this is astronomically unlikely; no special auth screen UX required for this case. Note: Convex indexes are not unique-constraint indexes — the retry loop is the only uniqueness guard.
- `claimReferralCode(code)` (public mutation, `convex/referrals.ts`) — takes `{ code: v.string() }`. Lookup chain: (1) `requireAuth(ctx)` returns `{ user, profile }` — `user.emotionalProfileId` is the referee's `refereeProfileId` (confirmed: `users` table has `emotionalProfileId: v.id("emotional_profiles")` at schema line 62). (2) Query `users.by_referral_code` index with the provided code → `referrer.emotionalProfileId` is `referrerProfileId`. Both `completeSession` and `completePath` are in `convex/sessions.ts` (confirmed: lines 245 and 184 respectively); `profileStats.updateAfterSession` is a separate scheduled mutation called via `ctx.scheduler.runAfter` from those mutations — not via `ctx.runMutation` — so it runs in a separate transaction, confirming the `sessionCount === 0` check in the parent transaction is always safe. Guards: (a) reject if `refereeProfileId === referrerProfileId` (self-referral); (b) reject if `referrals.by_referee.first()` returns a row for this `refereeProfileId` (one referral per user lifetime). Inserts `referrals` row with status `"pending"` — audit record only in v1, no UI action triggered on pending status. `by_code` index on the referrals table exists for analytics queries only — not used by any mutation in v1. Patches `emotional_profiles.referredByProfileId` on the referee's profile.
- `getMyReferralInfo()` (public query, `convex/referrals.ts`) — reads caller's `users` row for `referralCode`, reads `emotional_profiles` for `referralCount`, reads `preferences.referral.unlockedTiers`. Returns `{ code: user.referralCode ?? null, count: referralCount ?? 0, unlockedTiers: unlockedTiers ?? [] }`. The UI client hardcodes the full tier list `["first_referral", "three_referrals", "five_referrals"]` and checks each against `unlockedTiers` to determine earned/unearned display state. Queries cannot call mutations — backfill is NOT done inside this query. See backfill strategy in Dependencies.
- `backfillMyReferralCode()` (public mutation, `convex/referrals.ts`) — generates and sets `referralCode` on the caller's `users` doc if it is `undefined`. Called from a `useEffect` in the Settings "Refer a Friend" section when `getMyReferralInfo` returns `code: null`. Idempotent — no-ops if code already set.
- `completeReferral(refereeProfileId)` (internal mutation, `convex/referrals.ts`) — **wired in two places**: (1) `convex/sessions.ts` `completeSession` (exit path) and (2) `convex/sessions.ts` `completePath` (solo/peers path — only the `pathCompleted: true` branch, not the "confirmed but never path-selected" case). Scheduling strategy: **inside each mutation, add `const profile = await ctx.db.get(session.emotionalProfileId)` to bring the profile into scope, then only call `ctx.scheduler.runAfter(0, internal.referrals.completeReferral, { refereeProfileId: session.emotionalProfileId })` if `profile.sessionCount === 0`**. This read happens in the same mutation transaction, before `profileStats.updateAfterSession` increments the count. `completeReferral` itself does NOT re-check sessionCount.

  Inside `completeReferral`: (1) Query `referrals.by_referee.first()` using `refereeProfileId` — if no row exists, exit early (referee had no referrer). (2) If row status is already `"completed"`, exit early (idempotency guard for scheduler retries). (3) Patch row status to `"completed"`. (4) Read `referrerProfileId` from the row. (5) Get the referrer's `emotional_profiles` doc, increment `referralCount` (`(doc.referralCount ?? 0) + 1`). (6) Query `preferences` table via `by_profile` index using `referrerProfileId` to get the referrer's preferences doc. (7) Check milestone thresholds `[1, 3, 5]` against new count, patch `preferences.referral.unlockedTiers` if a new tier is crossed.

### Milestone unlock tiers

These unlocks are earned through referrals only — no payment, no premium tier. A user earns them by referring friends who complete their first session.

| Referrals | Unlock |
|-----------|--------|
| 1 | Ember color theme (warm amber/orange variant) |
| 3 | Extended voice vent time (10min/day → 15min) |
| 5 | New exclusive mirror tone (TBD name) unlocked permanently |

Unlock check: after `completeReferral` increments the count, check thresholds and patch `preferences.referral.unlockedTiers` if not already present. Surface milestone as a **toast** — triggered from the UI layer by watching `unlockedTiers` via `useQuery(api.referrals.getMyReferralInfo)`. Use a `useEffect` that compares `unlockedTiers.length` to a ref tracking the previous length; when length increases, fire the toast for each newly added tier. Use `unlockedTiers.length` comparison (not deep equality) — array reference changes every render. If the user is not on a screen with this query mounted when the referral completes (app backgrounded, etc.), the toast fires the next time they open a screen that mounts the query — v1 acceptable. PostHog event `referral_milestone_reached` fires at the same point, properties: `{ tier: "first_referral" | "three_referrals" | "five_referrals", referralCount: number }`.

**Tier gating in settings UI (referral-earned only, not premium):**
- Color theme: `unlockedTiers.includes("first_referral")` gates the Ember option in the theme picker. Unearned state shows the option with "Earn with 1 referral" — not locked behind payment, just not yet earned.
- Voice vent: the extended time unlock is enforced in `convex/vent.ts` (confirmed this file exists) — when checking the daily quota, read `preferences.referral.unlockedTiers` and apply 15 min/day if `"three_referrals"` is present, otherwise 10 min/day. No new field needed; the existing `preferences.referral.unlockedTiers` array is the source of truth. If the voice vent feature is not yet live when referral ships, the 3-referral row in the milestone display shows as "Coming soon" — the count still tracks, unlock applies retroactively when the feature ships.
- Mirror tone: `unlockedTiers.includes("five_referrals")` gates the new exclusive referral tone in the mirror tone picker. Unearned state shows "Earn with 5 referrals". This is the 6th tone — "Witnessed" already exists ungated and is left as-is.
  - **Pre-build gate**: the tone name and voice character must be decided before any code is written for the 5-referral tier. This is an explicit blocker — the schema literal, TONE_OPTIONS entry, and AI prompt cannot be written without it. Treat as a design prerequisite on the same level as copy finalisation.
  - **New tone deliverables** (once name is decided): (a) Add `v.literal("newToneName")` to the `mirrorTone` union validator in both `convex/schema.ts` (two locations, ~lines 240 and 395) and `convex/lib/validators.ts` — additive change, existing documents unaffected; (b) Add the tone to `TONE_OPTIONS` in `src/components/settings/mirror-tone-picker-dialog.tsx` with a `unlockedTiers` gate; (c) Write the AI prompt variant in `convex/ai/prompts/` following the existing tone prompt pattern.
  - Current tone picker UI can accommodate 6 tones without layout changes. Future premium tones will require a new picker design — separate task, out of scope.

**Deferred (v2+):**
Physical merch (t-shirts, branded items) and the 10-referral milestone are explicitly deferred. When merch ships, a redemption link is surfaced in settings for eligible users; shipping addresses are collected via an external form (Typeform/Printful) and never stored in Xolace.

**Post-auth code entry screen:**
- **Routing**: new file `src/app/(protected)/claim-code.tsx`. It is already called by the reflect screen. Once `referredByProfileId` is added to the schema, it will appear in this return automatically. Do not add a second query; read from the existing result. The `useEffect` runs when this query result transitions from `undefined` to a resolved value.
- **Gate** (checked in `(protected)/index` `useEffect`, runs only when profile is non-undefined): `!profile.referredByProfileId AND profile.sessionCount === 0 AND NOT localStore.referralCodePromptDismissed`
  - `!profile.referredByProfileId` handles both `undefined` (new optional field, existing rows) and falsy correctly — do not use `=== null`
  - `referredByProfileId` and `sessionCount` are both on `emotional_profiles` — same document, one read, no extra call
  - `localStore.referralCodePromptDismissed` is in the Zustand `TogglesSlice` in `src/store/store.ts`. Add field `referralCodePromptDismissed: boolean` (initial `false`) and a setter `setReferralCodePromptDismissed: (v: boolean) => void` to `TogglesSlice`. Include `referralCodePromptDismissed` in the `partialize` list alongside `notifNudgeDismissed`.
  - `sessionCount === 0` ensures existing users never see the prompt when the feature ships
- "Enter code" path: calls `claimReferralCode`. On success: sets `localStore.referralCodePromptDismissed = true`, fires PostHog `referral_code_entered`, navigates back. On failure: show inline error below the input field — "Invalid code" (code not found), "Already used" (referee already has a referrer), "Can't use your own code" (self-referral). Input remains editable; user can retry or tap Skip.
- "Skip" path: sets `localStore.referralCodePromptDismissed = true` immediately, no referral row created, navigates back.

### Share message / in-app copy principles

- Never use the word "Xolace" in the default share message text itself — the App Store link will naturally surface the app name in the Store listing, which is fine. The principle is about the message body not being a branded advertisement ("Download Xolace!") but a personal recommendation.
- In-app referee experience: "Someone who cares about you shared this with you." — no name shown
- Referral counter in settings: quiet, non-celebratory. "3 people you shared this with have joined." Not a streak, not a badge count.

## Resolved Decisions

1. **`referredBy` placement**: stored as `referredByProfileId` on `emotional_profiles` (not `users`). Consistent with privacy constraint — referral linkage stays at the profile level, not the auth-identity level. Matches the pattern where `sessionCount`, `currentStreak`, etc. all live on `emotional_profiles`.
2. **Late code entry**: N/A — the post-auth prompt is the only entry point. Recipients have the code before downloading the app. Skip is permanent. `sessionCount === 0` gates existing users out without an extra fetch — it's on the same `emotional_profiles` document as `referredByProfileId`.
3. **Fraud prevention v1**: one row per (referrer, referee) pair, enforced at mutation level. Self-referral guard included. Device-based dedup deferred to v2.
4. **"Completed" trigger**: first completed session by the referee (`sessions.status === "completed"` AND `emotional_profiles.sessionCount === 1`). Prevents fake signups from inflating counts.

## Success Criteria

- 15%+ of active users have shared their referral code within 30 days of launch
- 30%+ of referred users (who enter a code) complete their first session
- Referral-originated users show measurably higher 7-day retention vs. organic installs
- PostHog events:
  - `referral_code_copied` — fired in Settings "Refer a Friend" on copy button tap
  - `referral_code_shared` — fired in Settings and session-end on share sheet open
  - `referral_code_entered` — fired in claim-code screen on successful `claimReferralCode`
  - `referral_milestone_reached` — fired in UI when `unlockedTiers` length increases (via `getMyReferralInfo` query), properties: `{ tier, referralCount }`
