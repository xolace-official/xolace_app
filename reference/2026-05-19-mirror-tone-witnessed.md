# CEO Plan: Mirror Tone — Witnessed + Tone Discoverability

## Context

Users are reporting they want to feel *acknowledged* before the mirror reflects their emotions back with precision. The current 4 tones (adaptive, poetic, gentle, direct) all operate in precision-first mode. The articulator prompt has a half-measure: "In some rare cases you can give acknowledgement as part of the mirror but only if it feels like a natural extension" — but that leaves it entirely to model judgment, which isn't consistent enough to satisfy the user need.

Additionally, tones are invisible to users during sessions — there's no indicator on the mirror screen telling them what tone is active, and no discoverability path for first-time users.

## Accepted Scope

### 1. "Witnessed" tone
New 5th tone option in settings. Mirror leads with brief recognition, then delivers precision reflection.

### 2. Tone indicator badge
Small badge/chip under "The Mirror" label. **Only shown for non-adaptive tones** (poetic, gentle, direct, witnessed). When `mirrorTone` is `"adaptive"` OR `undefined` (loading state) — no badge. This is intentional: the one-time tip banner handles first-use discoverability; the badge confirms a conscious non-default choice.

### 3. One-time tip banner
**Copy spec:**
- Headline: none (banner is compact)
- Body: `"You can change how the mirror reflects. Explore different tones."`
- CTA: `"Adjust tone"` (opens `MirrorTonePickerDialog` directly — see Implementation Note below)
- Dismiss: `✕` icon button

**Persistence:** `toneTipSeen: boolean` in Zustand `TogglesSlice`. **This is a global one-time flag** — once dismissed, the banner never shows again regardless of how many sessions the user has, or whether the app is restarted. Initial value: `false`. Must be added explicitly to the `partialize` allowlist in `store.ts`.

**Implementation Note on deep-link (replaces router.push approach):** Rather than `router.push('/settings')` (which risks breaking the in-session state machine), open `MirrorTonePickerDialog` directly from the banner via a local `useState` boolean in `mirror-state.tsx`. This is safer: no navigation, no remount risk, user stays in session, picks a tone, sees the indicator update immediately. The settings route still contains the tone picker for normal settings access.

### 4. PostHog events (4 total — user-initiated only, not on mount)
- `tone_tip_shown` — fires once on first render of the banner (guarded by `!toneTipSeen`). Impression denominator.
- `tone_tip_dismissed` — fires on `✕` press. % who ignore the tip.
- `tone_tip_tone_selected` — fires when user selects a tone via the banner-triggered picker. Conversion: tip → actual change.
- `tone_changed` — fires in the settings picker `onSelect` handler. All tone changes, including those from normal settings access.

### 5. Gave Up state tone hint
One-liner + link rendered **regardless of current tone**. Copy: `"Explore different mirror tones →"` (not "try a *different*" tone — "explore" is neutral and works whether user is on adaptive or a named tone). Links to `router.push('/settings')` (settings navigation from gave-up state is low-risk — no active session being interrupted).

## Schema Backward Compatibility Note
Adding `"witnessed"` to `mirrorTone v.union()` is additive. Existing stored values are unaffected. Unknown values already fall through to the `default` case in `getToneInstructions()` which returns adaptive behavior. **Rollback caveat:** if `"witnessed"` is later removed from the union, any preference document with `mirrorTone: "witnessed"` would fail schema validation on read. Before removing, run a migration to set those records back to `"adaptive"`.

## Witnessed Tone Prompt Specification

Add this case to `getToneInstructions()` in `articulator.ts`:

```typescript
case "witnessed":
  return `Begin with one brief line of recognition that names the emotional weight the user is carrying — before mirroring it back. The recognition should feel like a human noticing, not a therapist validating. It names the weight, not the content.

  Good recognition lines: "That's a lot to carry." / "Something heavy landed." / "That kind of tired goes deeper than sleep." / "That took something out of you."
  Bad recognition lines: "I hear you." / "I understand." / "That must be hard." / "I see you." (too direct, clinical, or clichéd)

  After the recognition line, deliver the precision mirror exactly as the other tones do (1-3 sentences). Total response: 2-4 sentences.

  Rules:
  - Recognition must NOT repeat the user's words — it names the weight, the mirror captures the shape
  - Do not moralize or comfort ("it'll get better", "you're not alone")
  - Recognition line is always first, always one sentence
  - If the user's input is short (1-2 sentences), the recognition + mirror together should still be 2-3 sentences max`;
```

**Default case note:** `getToneInstructions()` already has a `default:` case returning adaptive behavior. Unknown values are safe.

## Hook for mirrorTone in mirror-state.tsx

```typescript
// Inside MirrorState component:
const { mirrorTone } = useSettings();
// undefined (loading) treated same as "adaptive" — no badge. Intentional.
const showToneBadge = mirrorTone && mirrorTone !== "adaptive";
const toneLabel = mirrorTone ? mirrorTone.charAt(0).toUpperCase() + mirrorTone.slice(1) : "";
```

**Subscription safety:** `useSettings()` adds a Convex reactive subscription inside `MirrorState`. This is fine — the subscription only affects the badge display and the tip banner state. `MirrorState` has no `useEffect` that dispatches state machine actions based on preference changes. The `mirrorTone` value is display-only in this component.
