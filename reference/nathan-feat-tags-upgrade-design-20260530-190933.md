# Design: Texture Word Sets — Categorized Emotional Vocabulary

## Problem Statement

The current 8 texture words (heavy, tight, foggy, buzzing, empty, scattered, numb, raw) feel static after extended use. The root cause is **fit**, not volume — the current set covers only somatic/flat/heavy emotional states. A user who feels hopeful, frustrated, tender, or activated has no words that match. The fix is multiple curated word sets organized by emotional register, with an inline tab switcher on the idle screen so the user can shift vocabulary without friction.

## What Makes This Cool

The app already proved the concept invisibly — Night Mode auto-swaps to a completely different word set (wide awake, can't turn it off, spiraling, aching...) based on time. This feature makes that insight explicit and user-controlled: the right words for how you feel right now, one tap away.

## Constraints

- OTA-safe (no native modules, no native rebuild required)
- Local only — no Convex writes, no backend changes
- Words hardcoded for now (dynamic/AI-generated word sets are a future evolution)
- Active set stored in Zustand, persisted across sessions
- Night Mode always overrides to `NIGHT_TEXTURE_WORDS` — night mode is not a user-selectable set
- Tab labels must be 4-7 chars max (inline display, small screen); "Charged" at 7 chars is the hard limit — verify on SE-sized hardware
- No clinical language — no "Depressed", "Anxious disorder", etc.
- React Compiler is enabled — no manual useMemo/useCallback

## Premises

1. **Root cause is emotional fit, not word volume** — 8 well-chosen words for a specific emotional register is better than 40 random words. Confirmed.
2. **User explicitly selects the set (v1)** — auto-suggestion (via prior session tone or time heuristics) is the right long-term direction, but explicit selection validates whether different sets get used at all before investing in inference logic.
3. **Inline tab switcher, not a modal picker** — the set selector lives right above the word cloud so the user sees their options alongside the words themselves. No extra navigation required.
4. **Night Mode is separate and not user-selectable** — it auto-activates between 10pm-4am and always overrides whatever set the user picked. Night words stay in `night-copy.ts`.
5. **4 sets is the right starting count** — enough variety to cover the major emotional registers, few enough that the tab row fits without wrapping on a standard phone screen.

## Word Sets

### Flat (default — somatic/unnameable)
The current set. Body sensations, textures, the "can't name it" state.
```
heavy, tight, foggy, buzzing, empty, scattered, numb, raw
```

### Tender (grief/loss/sadness register)
Quiet pain, absence, the ache of something missing.
```
aching, hollow, longing, bereft, weeping inside, closing off, tender, heavy-hearted
```

### Bright (warmth/hope/gratitude register)
Expansion, lightness, things that feel like yes.
```
warm, open, grateful, hopeful, full, glowing, clear, lifted
```

### Charged (activated/anxious/frustrated register)
High-energy, can't-settle, something-about-to-break states.
```
racing, bracing, on-edge, restless, wound up, grinding, spinning, pressed
```

## Approaches Considered

### Approach A: Explicit set picker (bottom label or chip)
A small tappable label ("Change set →") above the word cloud opens a bottom sheet or inline label row. User picks a set, closes the picker, then sees the words.

- Effort: XS–S
- Risk: Low
- Con: Adds a pre-classification step before the user sees any words

### Approach B: Inline tab pills above the word cloud (CHOSEN)
A compact horizontal `TabGroup`-style row sits between the "Or just tap what feels close:" label and the word cloud. Labels: `Flat | Tender | Bright | Charged`. Tapping a tab swaps the word list in place with a brief fade transition. Active set stored in Zustand.

- Effort: S (human: ~4h / CC: ~30min)
- Risk: Low
- Con: Adds one visual row to the idle screen; tab labels must be short

### Approach C: Rotating pool — no categories
Expand to 30-40 words, randomly sample 8-10 per session. No UI change.

- Effort: XS
- Risk: Low
- Con: Doesn't solve fit — hopeful words mixed with heavy words at random

## Recommended Approach

**Approach B — inline tab pills.**

The tab row appears directly above the word cloud so the user sees the sets alongside the words. No pre-classification step, no modal, no extra navigation. One tap = set swap = words change. This mirrors the clarity of Night Mode but makes it user-controlled and persistent.

The set name (`textureSetId`) is stored in a new Zustand `preferences` slice (see Implementation Notes — State). On each render, `IdleState` reads the active set name and resolves it to the word array from a static map. Night Mode check happens before set resolution: if `isNight` is true, NIGHT_TEXTURE_WORDS always wins. Unknown/invalid persisted values fall back to `'flat'`.

## Implementation Notes

### State

Add a new `preferences` slice to `src/store/store.ts` (parallel to `TogglesSlice`, `ThemeSlice`, etc.):

```ts
type PreferencesSlice = {
  textureSetId: TextureSetId;
  setTextureSetId: (id: TextureSetId) => void;
};
```

Default: `textureSetId: 'flat'`.

Add to `partialize` to persist across sessions:
```ts
// in the partialize function
textureSetId: s.textureSetId,
```

No migration needed — `textureSetId` is a new key. When absent from persisted state (first run after upgrade), Zustand resolves it to the default `'flat'`. Defensive resolution at `IdleState` render time handles corrupted storage (wrong type, `null`, removed set name):
```ts
const TEXTURE_SET_IDS: TextureSetId[] = ['flat', 'tender', 'bright', 'charged'];
const safeSetId: TextureSetId =
  typeof textureSetId === 'string' && TEXTURE_SET_IDS.includes(textureSetId as TextureSetId)
    ? (textureSetId as TextureSetId)
    : 'flat';
```
Also: before adding `CLEAR_TEXTURES` to `reflection-reducer.ts`, grep for the action name to confirm it doesn't already exist under a different name.

Implement as a horizontal `ScrollView` of `PressableFeedback` pills (not HeroUI `Tabs` — HeroUI Tabs is for page-level navigation; these are lightweight filter pills). The active pill uses `text-accent` and a subtle `bg-accent/10 border-accent/30` surface; inactive pills use `text-foreground/40`. No scrolling needed at 4 sets — a plain `View` with `flex-row` is sufficient.

`idle-state.tsx`:
- Remove the inline day word list constant (currently `DAY_TEXTURE_WORDS` at `idle-state.tsx:35` — grep to confirm name before removing)
- Import and render `<TextureSetTabs activeSet={textureSetId} onSelect={handleSetChange} />`
- Word resolution: `TEXTURE_SETS[resolvedSetId] ?? TEXTURE_SETS.flat` where `resolvedSetId` is the validated store value
- `handleSetChange`: dispatch `CLEAR_TEXTURES` first, then call `setTextureSetId(id)`, then play `playSoftPress` haptic

**`TagGroup` key stability:** Add `key={resolvedSetId}` to the `<TagGroup>` wrapper. This forces HeroUI Native to remount the component when the set changes, clearing any internal selection state. This is the correct pattern because `TagGroup` may cache selection internally.

```tsx
<TagGroup key={resolvedSetId} selectionMode="multiple" ...>
```

### Animation

When the active set changes, words should fade out then fade in — not a simultaneous crossfade (two overlapping `TagGroup`s is unnecessary complexity for 8 words).

`idle-state.tsx` needs three local state variables to drive this:
```ts
const [wordsVisible, setWordsVisible] = useState(true);
const [pendingSetId, setPendingSetId] = useState(textureSetId);  // from Zustand
const [resolvedSetId, setResolvedSetId] = useState(textureSetId); // drives word list
```

Handler ordering (all in `handleSetChange`):
```ts
const handleSetChange = (id: TextureSetId) => {
  dispatch({ type: 'CLEAR_TEXTURES' });  // 1. clear selections synchronously
  setTextureSetId(id);                   // 2. write to Zustand immediately (persistence)
  setPendingSetId(id);                   // 3. stage incoming set for animation
  setWordsVisible(false);               // 4. trigger fade-out
  playSoftPress();                       // 5. haptic
  // resolvedSetId and wordsVisible=true fire in onTransitionEnd (step 6-7)
};
```

Fade animation:
```tsx
<EaseView
  animate={{ opacity: wordsVisible ? 1 : 0 }}
  transition={{ type: 'timing', duration: wordsVisible ? 200 : 150 }}
  onTransitionEnd={({ finished }) => {
    if (finished && !wordsVisible) {
      setResolvedSetId(pendingSetId);  // 6. swap word list while hidden
      setWordsVisible(true);           // 7. trigger fade-in
    }
  }}
>
  <TagGroup key={resolvedSetId} ...>
```

`setTextureSetId` (Zustand) fires immediately in the handler — not in `onTransitionEnd` — so the preference is persisted even if the user backgrounds the app mid-animation.

**EaseView risk:** This pattern assumes `EaseView` fires `onTransitionEnd({ finished })` reliably on opacity exit. Verify this against `react-native-ease`'s API before committing. If `onTransitionEnd` is not supported or doesn't receive `{ finished }`, use a `runOnJS` callback inside a Reanimated `withTiming` instead. The rest of the pattern holds.

Per project feedback: always use `onTransitionEnd`, never `setTimeout`.

### Night Mode Interaction

```
isNight === true  →  always use NIGHT_TEXTURE_WORDS, hide tab row
isNight === false →  show tab row, use TEXTURE_SETS[resolvedSetId]
```

The tab row is **conditionally rendered** (not animated out) — `{!isNight && <TextureSetTabs ... />}`. Night Mode activates at a session boundary (user opens the app between 10pm–4am), not mid-session, so no in-app transition animation is needed for the tab row.

`textureSetId` in Zustand is **never touched** by Night Mode. When Night Mode exits, the user's previously selected set restores automatically because `textureSetId` is still in the store.

### Multi-word Tags

"weeping inside" and "heavy-hearted" are 2-word tags. `TagGroup.Item` handles multi-word content fine. Use `min-w-fit` (not `min-w-18` — `min-w-18` is non-standard in Tailwind/Uniwind and may not resolve). The existing `min-w-18` on single-word tags should be audited — if it works today it's likely a custom token; if not, swap to `min-w-20`. Verify "weeping inside" renders without truncation on SE (320px wide screen).

## Resolved Decisions

1. **Persist `textureSetId`** — yes. User's last-used set restores on next open. Feels like a preference, not a session state.

2. **Slice placement** — new `preferences` slice in `store.ts`, separate from `toggles`. Cleaner separation of "feature on/off" vs "user preference values."

3. **Haptic** — `playSoftPress`. Lighter than `playTextureSelect` (word tap), distinguishes set switch from word selection.

4. **Set label** — keep "Flat", not "Default". "Flat" carries meaning (flat affect) and matches the Xolace language register.

5. **Ghost selections on set switch** — handled by dispatching `CLEAR_TEXTURES` synchronously before triggering the animation. See Component Changes for ordering guarantee.

6. **`Activity` from React** — not needed. We want state to clear on set switch, not preserve it. `key={resolvedSetId}` on `TagGroup` handles remount correctly.
